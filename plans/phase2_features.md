# 阶段 2 — 功能完备（OCR 与导入导出）

目标（时长：6 周）
- 集成 OCR 流程并完善导入/导出与分析能力，提升数据输入效率与可视化能力。

周计划
- 周 1（OCR 架构与 stub）：设计 OCR 接口抽象（`OcrEngine`），实现 ML Kit stub，准备测试用图片/文本样例。
- 周 2（Parser 策略）：实现 ReceiptParser 框架与策略（`DefaultParser`、`YonghuiParser`、`HemaParser`）；实现策略切换接口并添加单元测试。
- 周 3（Receipt 流）：实现拍照/选择图片 → OCR → Parser → 编辑列表 → 写入 DB 的完整流程；编写端到端集成测试（可用模拟数据替代 ML Kit）。
- 周 4（导入/导出）：实现 `.mypd`（gzip+json）导出与导入，并验证数据完整性和版本兼容；添加导入冲突解决策略（覆盖/合并/跳过）。
- 周 5（图表与仪表盘）：实现价格趋势图、商店比价表、仪表盘统计；保证 SQL 聚合与前端数据一致。
- 周 6（QA 与优化）：修复集成测试发现的问题、UI 可用性调整、补充测试与文档。

交付物
- OCR 抽象与实现（含 stub 与集成说明）。
- ReceiptParser 多策略实现与测试样例。
- 完整的 Receipt 导入识别写入流程与集成测试。
- 导入/导出实现与数据迁移说明。
- 增强的图表组件与数据校验。

验收标准
- OCR 能输出 `ocrRawText` 并被 Parser 转为 `ReceiptItem` 列表。
- 导入/导出能完整恢复数据并通过完整性校验。
- 图表与聚合查询结果一致且响应可接受。

AI 生成提示范例
- "实现 `ocr/OcrEngine.kt` 抽象接口和 ML Kit stub，返回接口文件、stub 实现、使用示例与测试说明。"

风险与缓解
- OCR 真实环境表现不稳定：先走 stub 与规则解析；后期可增加模型或云服务作为可选项。
- 导入大文件性能：对导入做流式解析与批量写入，阶段 3 进行性能验证。
